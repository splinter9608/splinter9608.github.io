EC2는 서버다. 맞는 말이다. AWS에서 가상 서버를 빌려 쓰는 서비스.

근데 이 한 줄 설명이 오히려 EC2를 제대로 이해하는 데 방해가 된다. "서버를 빌린다"고 생각하는 순간, 선택지가 좁아진다. 어떤 서버를 빌릴지, 얼마나 빌릴지, 어떻게 빌릴지 — 이 결정들이 곧 아키텍처가 된다.

**EC2를 제대로 쓰려면 서버를 빌리는 게 아니라 건물을 설계한다고 생각해야 한다.**

이번 글에서는 1) EC2의 핵심 구성 요소, 2) 인스턴스 유형과 선택 기준, 3) 구매 옵션별 비용 전략을 살펴봅니다.

---

## 건물을 짓기 전에 설계도가 필요하다

건물을 짓는다고 생각해보자. 가장 먼저 필요한 건 설계도다. 어떤 용도의 건물인지, 기본 구조가 어떻게 생겼는지.

EC2에서 이 설계도 역할을 하는 게 **AMI(Amazon Machine Image)**다.

> **AMI** — EC2 인스턴스를 시작하기 위한 템플릿. 운영체제(OS), 설치된 소프트웨어, 초기 설정이 패키지로 묶여 있다.

AMI 없이는 EC2 인스턴스를 만들 수 없다. 어떤 AMI를 고르느냐에 따라 인스턴스가 "Amazon Linux인지, Ubuntu인지, Windows인지"가 결정된다.

설계도는 세 가지 출처에서 구할 수 있다:

> - **AWS 공식 AMI** — Amazon Linux, Ubuntu, Windows 등. 빠르게 시작하기에 가장 좋다.
> - **AWS Marketplace** — 소프트웨어가 미리 설치된 AMI. WordPress, Jenkins, SAP 등 구성된 채로 시작할 수 있다.
> - **내 AMI** — 직접 구성한 인스턴스를 AMI로 저장해두면 동일한 환경을 여러 대 찍어낼 수 있다.

특히 마지막이 중요하다. **직접 만든 AMI가 있으면, 같은 설정을 가진 서버를 몇 초 만에 100대도 뽑아낼 수 있다.** ASG(Auto Scaling Group)에서 자동으로 서버를 늘릴 때 이 AMI를 기반으로 찍어낸다.

## 건물 크기와 용도를 고른다 — 인스턴스 유형

설계도(AMI)가 준비됐으면 다음 결정은 건물의 크기와 용도다. EC2에서는 이를 **인스턴스 유형**이라고 한다.

인스턴스 유형은 CPU, 메모리, 네트워크 성능의 조합이다. 이름에 규칙이 있다.

> `t3.micro`
> - `t` — 인스턴스 패밀리 (용도)
> - `3` — 세대 (숫자가 높을수록 최신)
> - `micro` — 크기 (nano < micro < small < medium < large < xlarge < ...)

패밀리별로 쓰임새가 다르다:

> - **t 계열 (범용)** — 소규모 웹 서버, 개발/테스트 환경. 평소엔 조용히 있다가 필요할 때만 CPU를 터뜨리는 구조 (버스트 가능). 프리 티어 `t2.micro`가 여기 속한다.
> - **c 계열 (컴퓨팅 최적화)** — 고성능 웹 서버, 배치 처리, 머신러닝 추론. CPU 성능이 중요할 때.
> - **r 계열 (메모리 최적화)** — 대용량 인메모리 데이터베이스, 실시간 분석. RAM이 많아야 할 때.
> - **i 계열 (스토리지 최적화)** — IOPS가 중요한 데이터베이스, 데이터 웨어하우스.

처음엔 전부 `t3.micro`나 `t3.small`로 시작해도 된다. 인스턴스 유형은 나중에 바꿀 수 있다. 인스턴스를 중지하고 유형만 변경하면 끝이다.

**어떤 유형을 써야 할지 모르면, 병목이 어디서 오는지를 먼저 파악하는 게 순서다.** CPU가 100%인지, 메모리가 부족한지, 디스크 I/O가 느린지. 그 병목에 맞는 패밀리를 고르면 된다.

## 건물 계약 방식이 비용을 결정한다 — 구매 옵션

같은 인스턴스라도 어떻게 계약하느냐에 따라 비용 차이가 크다. EC2에는 네 가지 구매 방식이 있다.

### On-Demand — 월세
필요할 때 시작하고, 쓴 만큼만 낸다. 시간 단위 혹은 초 단위 과금. **유연하지만 가장 비싸다.** 트래픽을 예측하기 어렵거나, 실험·개발 중인 상황에 적합하다.

### Reserved Instance — 전세 (1년~3년 계약)
1년 또는 3년 약정을 걸면 On-Demand 대비 최대 72% 할인된다. **서비스가 안정화됐고 베이스라인 서버 수가 명확할 때 쓴다.** 항상 켜져 있어야 하는 서버라면 Reserved가 훨씬 경제적이다.

### Spot Instance — 경매
AWS의 남는 컴퓨팅 자원을 경매 방식으로 낙찰받는다. On-Demand 대비 최대 90% 저렴하다. 단, AWS가 자원을 회수하면 2분 경고 후 종료된다.

**인터럽트(중단)를 허용할 수 있는 작업에만 쓴다.** 배치 처리, 이미지 렌더링, 데이터 분석처럼 중간에 끊겨도 이어 할 수 있는 작업이 여기 해당한다. 웹 서버처럼 항상 떠 있어야 하는 서버엔 절대 쓰면 안 된다.

### Dedicated Host — 건물 통째로 임대
물리 서버 전체를 내가 독점으로 사용한다. 다른 AWS 고객과 하드웨어를 공유하지 않는다. 규정상 멀티테넌트 환경을 쓸 수 없는 금융, 의료 같은 컴플라이언스 요구사항이 있을 때 쓴다. 가장 비싸다.

### Savings Plans — 유연한 장기 계약
Reserved Instance가 특정 인스턴스 유형에 묶인다면, Savings Plans는 "1년 동안 시간당 $X 이상 쓸게요"라는 금액 약정이다. 어떤 인스턴스 유형으로 바꿔도 할인이 유지된다. On-Demand 대비 최대 66% 절감 가능하다.

**인스턴스 유형을 바꿀 가능성이 있다면 Reserved보다 Savings Plans가 낫다.** 서비스가 성장하면서 t3에서 c5로 넘어가는 식의 변화를 감당할 수 있다.

## 건물 주소가 재시작마다 바뀐다 — Elastic IP

EC2 인스턴스는 시작할 때마다 새로운 퍼블릭 IP를 받는다. 재시작하면 주소가 바뀐다. 서버 주소를 직접 공유했거나 DNS에 IP를 하드코딩해뒀다면, 재시작 순간 연결이 끊긴다.

**Elastic IP**는 내 계정에 고정된 퍼블릭 IP다. 인스턴스에 붙이면 재시작해도 주소가 유지된다.

그런데 실제로 Elastic IP를 직접 쓰는 경우는 많지 않다. 인스턴스 앞에 ELB를 두면 ELB DNS 이름으로 접근하기 때문에 IP가 바뀌는 문제 자체가 없어진다. **Elastic IP는 ELB 없이 단일 서버를 고정 주소로 운영해야 할 때 쓰는 임시방편에 가깝다.**

> 참고: Elastic IP는 인스턴스에 붙여두면 무료지만, 인스턴스 없이 할당만 해두면 요금이 발생한다.

## 건물 완공 직후 자동으로 할 일 — User Data

인스턴스가 처음 시작될 때 자동으로 실행할 스크립트를 넣을 수 있다. 이게 **User Data**다.

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
echo "<h1>서버가 떴습니다</h1>" > /var/www/html/index.html
```

건물 완공 직후 "입주 청소 + 가구 배치"를 자동으로 해주는 것과 같다. 직접 SSH로 들어가서 설치할 필요 없이, 인스턴스가 뜨자마자 원하는 상태로 만들어준다.

**AMI + User Data 조합이 EC2 자동화의 기본이다.** AMI로 기본 환경을 찍어내고, User Data로 애플리케이션별 커스텀 설정을 한다.

## 건물 열쇠와 담장 — Key Pair와 보안 그룹

서버에 SSH로 접속하려면 열쇠가 필요하다. EC2에서는 이를 **Key Pair**라고 한다. 공개키는 AWS에, 개인키(.pem 파일)는 내 로컬에 저장된다. **개인키를 잃으면 그 인스턴스에 다시 들어갈 방법이 없다.** 잘 보관해야 한다.

담장 역할은 **보안 그룹(Security Group)**이 한다. 어떤 포트로 들어오는 트래픽을 허용할지 규칙을 정한다. 웹 서버라면 80(HTTP), 443(HTTPS), SSH라면 22번 포트를 열어둔다.

보안 그룹은 인스턴스에 붙이는 방화벽이다. 기본은 모두 닫혀 있고, 명시적으로 허용한 것만 통과한다.

## 서버를 직접 늘리지 않아도 되는 이유 — ASG

트래픽이 갑자기 몰리면 서버를 더 띄워야 한다. 근데 그걸 사람이 직접 하면 늦다.

**ASG(Auto Scaling Group)**는 정해진 기준에 따라 자동으로 EC2 인스턴스를 늘리고 줄인다. CPU 사용률이 70%를 넘으면 서버를 두 대 더 뽑고, 트래픽이 빠지면 다시 줄인다.

ASG는 세 가지 숫자로 동작한다:

> - **최소(Min)** — 항상 이 대수 이상은 유지한다
> - **최대(Max)** — 이 대수 이상은 절대 늘리지 않는다
> - **원하는 수(Desired)** — 현재 목표 대수

ELB와 ASG를 함께 쓰면, 서버가 자동으로 늘어나더라도 ELB가 새 서버를 감지해서 트래픽을 자동으로 넣어준다. 사람이 개입할 필요가 없다.

## 전체 흐름으로 보면

설계도(AMI)를 고르고 → 크기(인스턴스 유형)를 결정하고 → 계약 방식(구매 옵션)을 택하고 → 완공 후 자동 설정(User Data)을 넣는다. 보안 그룹으로 담을 치고, 열쇠(Key Pair)로 잠근다. 트래픽이 늘면 ASG가 알아서 같은 설계도로 건물을 더 짓는다.

**서버를 빌리는 게 아니라 건물을 설계하는 것이다.** 처음엔 단칸방 하나로 시작해도 된다. 중요한 건 언제든 같은 설계도로 100채를 찍어낼 수 있는 구조를 만들어두는 것이다.

건물이 준비됐으면, 그 앞에 안내데스크를 세울 차례다.
