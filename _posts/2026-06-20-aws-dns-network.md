브라우저에 google.com을 입력하면 구글이 뜬다. 당연하다.

근데 이게 당연한 게 아니다. 컴퓨터는 "google.com"이라는 이름을 모른다. **컴퓨터가 아는 건 숫자뿐이다.** 그 숫자를 IP 주소라고 한다.

그럼 google.com을 쳤을 때 실제로 무슨 일이 일어나는 걸까. 이게 AWS Route 53을 이해하는 출발점이고, 네트워크 전반을 이해하는 기본기다.

이번 글에서는 1) IP 주소가 뭔지, 2) 포트와 프로토콜이 왜 있는지, 3) 도메인 이름 구조, 4) DNS가 어떻게 작동하는지를 살펴봅니다.

---

## IP 주소 — 인터넷의 집 주소

편지를 보내려면 주소가 필요하다. 인터넷도 마찬가지다. 인터넷에 연결된 모든 기기에는 고유한 주소가 있다. 그게 IP 주소다.

> **IP 주소(Internet Protocol Address)** — 인터넷에 연결된 기기를 식별하는 숫자 주소.

IP 주소는 두 종류가 있다.

- **IPv4**: `172.217.18.36` 형태. 점(.)으로 구분된 숫자 4개. 약 43억 개의 주소를 만들 수 있다.
- **IPv6**: `2001:0db8:85a3::8a2e:0370:7334` 형태. 훨씬 길다. IPv4 주소가 부족해져서 나온 후속 규격.

지금은 대부분 IPv4를 쓴다. 43억 개면 충분할 것 같지만, 인터넷에 연결된 기기가 전 세계에 수백억 개가 넘으면서 고갈됐다. 그래서 IPv6로 전환이 진행 중이다.

**IP 주소는 인터넷의 집 주소다.** google.com도 결국 어딘가의 IP 주소를 갖고 있다.

---

## 포트 — 집 안의 방 번호

IP 주소로 집을 찾았다. 근데 집 안에 방이 여러 개라면?

하나의 서버는 여러 서비스를 동시에 운영할 수 있다. 웹 서비스도 하고, 이메일도 받고, 파일 전송도 한다. 포트가 그 구분을 한다.

> **포트(Port)** — IP 주소 안에서 특정 서비스를 구분하는 번호. 0~65535 사이의 숫자.

자주 쓰는 포트는 약속처럼 고정돼 있다.

- **80**: HTTP (일반 웹)
- **443**: HTTPS (암호화된 웹)
- **22**: SSH (서버 원격 접속)
- **3306**: MySQL (데이터베이스)

브라우저에서 `http://google.com`을 치면 사실 `http://google.com:80`과 같다. 포트 번호를 안 써도 HTTP면 80, HTTPS면 443으로 자동으로 간다.

**포트는 집 안의 방 번호다.** 같은 주소(IP)에서 어떤 서비스를 쓸지 구분해준다.

## 프로토콜 — 편지 봉투에 쓰는 규칙

편지를 보낼 때 아무렇게나 쓰면 안 된다. 받는 사람 주소, 보내는 사람 주소, 우표 — 정해진 형식이 있다.

인터넷에서 데이터를 주고받을 때도 형식이 있다. 그게 프로토콜이다.

> **프로토콜(Protocol)** — 데이터를 주고받는 방식의 약속. 송신자와 수신자가 같은 규칙으로 통신한다.

자주 보이는 것들:

- **HTTP**: 웹 페이지를 주고받는 규칙. 암호화 없음.
- **HTTPS**: HTTP에 암호화(TLS)가 더해진 버전. 지금 대부분의 웹사이트가 이걸 쓴다.
- **TCP/IP**: 인터넷 통신의 기반이 되는 핵심 프로토콜.

주소(IP) + 방 번호(포트) + 규칙(프로토콜)이 합쳐지면 인터넷 통신이 된다.

## 도메인 이름 — 주소 대신 이름으로

`172.217.18.36`을 외워서 구글에 가는 사람은 없다. 그래서 사람이 기억하기 쉬운 이름을 붙인 게 도메인이다.

도메인에는 계층 구조가 있다. `www.google.com`을 예로 들면:

```
www        .google      .com
서브도메인   2차 도메인    최상위 도메인(TLD)
```

- **TLD(Top Level Domain)**: 가장 오른쪽. `.com`, `.net`, `.org`, `.kr` 등. 인터넷 최상위 분류.
- **2차 도메인(SLD)**: TLD 바로 왼쪽. `google`, `amazon`, `naver` 같은 브랜드 이름.
- **서브도메인**: 2차 도메인 왼쪽. `www`, `mail`, `blog` 등. 같은 도메인 안에서 서비스를 구분.

`www.google.com` 전체를 **FQDN(Fully Qualified Domain Name)** 이라고 부른다. 완전한 도메인 이름.

[이미지: www.google.com 도메인 구조를 계층별로 나눈 그림]

**도메인은 사람을 위한 이름이고, IP는 컴퓨터를 위한 주소다.** 둘을 연결해주는 게 DNS다.

## DNS — 인터넷의 전화번호부

google.com을 치면 어떻게 `172.217.18.36`으로 이어질까.

전화번호부를 생각하면 된다. 이름은 알지만 번호를 모를 때, 전화번호부에서 이름으로 번호를 찾는다. DNS가 정확히 이 역할이다.

실제 조회 과정은 이렇다:

> 1. 브라우저가 `google.com`을 요청한다
> 2. 내 컴퓨터가 **로컬 DNS 서버**에 묻는다 (보통 인터넷 통신사가 제공)
> 3. 로컬 DNS가 캐시에 없으면 **Root DNS**에 묻는다 → `.com`을 담당하는 서버가 어딘지 알려준다
> 4. **TLD DNS**에 묻는다 → `google.com`의 네임 서버가 어딘지 알려준다
> 5. **구글 네임 서버**에 묻는다 → `172.217.18.36` 반환
> 6. 브라우저가 그 IP로 접속한다

[이미지: DNS 조회 과정 흐름도 — 브라우저 → 로컬DNS → Root → TLD → 네임서버]

한 번 조회한 결과는 캐시에 저장된다. 그래서 두 번째 접속부터는 빠르다. 이 캐시 유지 시간을 **TTL(Time To Live)** 이라고 한다. 캐시 편에서 본 그 TTL이다.

---

## Domain Registrar vs DNS 서비스

여기서 헷갈리는 부분이 하나 있다.

도메인을 사는 곳(Registrar)과 DNS 설정을 관리하는 곳(DNS 서비스)은 다를 수 있다.

> **Domain Registrar** — 도메인 이름을 구입하고 등록하는 서비스. GoDaddy, 가비아 등.
>
> **DNS 서비스** — 도메인과 IP를 연결하는 레코드를 관리하는 서비스. AWS Route 53 등.

GoDaddy에서 도메인을 사도, DNS 레코드는 Route 53에서 관리할 수 있다. Registrar에서 "네임 서버"를 Route 53 주소로 바꿔주면 된다.

**도메인 구입처와 DNS 관리 서버는 꼭 같을 필요가 없다.** AWS 쓴다고 Route 53에서 도메인을 살 필요는 없다는 뜻이다.

---

google.com을 치는 그 0.몇 초 사이에 전화번호부를 뒤지고, 집 주소를 찾고, 방 번호를 확인하고, 정해진 규칙으로 대화를 시작하는 일이 벌어진다.

우리는 그걸 인식하지 못한다. 인터넷이 그만큼 잘 설계돼 있기 때문이다.
