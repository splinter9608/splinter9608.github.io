일단 되게 만들고 나중에 고치면 된다. 개발 입문 시절에 많이 듣는 말이다.

틀린 말은 아니다. 근데 "나중에 고친다"는 게 생각보다 훨씬 비싼 작업이다. 특히 화면 하나에 모든 게 하드코딩돼 있으면, 뭔가 하나 바꾸려고 할 때 파일 전체를 다 뒤져야 한다. 같은 색상 코드가 37군데에 흩어져 있다는 걸 그때서야 알게 된다.

**하드코딩의 진짜 비용은 처음 만들 때가 아니라 두 번째로 건드릴 때 나온다.**

이번 글은 화면 하나에 전부 때려박혀 있던 코드를 3계층으로 분리한 리팩토링 경험을 정리한 것이다. 1) 왜 하드코딩이 문제인지, 2) 3계층 구조가 무엇인지, 3) 분리했을 때 실제로 무엇이 달라지는지를 살펴본다.

---

## 하드코딩의 문제는 처음엔 안 보인다

화면 하나를 만들 때 하드코딩은 빠르다. 색상도 직접 넣고, 폰트 크기도 직접 넣고, 버튼 스타일도 그때그때 만든다. 뭔가 만들어지는 속도가 빠르니까 진행되는 느낌이 든다.

문제는 두 번째 화면부터다.

비슷한 버튼이 필요해서 첫 번째 화면에서 복사해온다. 세 번째 화면도 마찬가지. 어느 순간 같은 버튼 코드가 화면마다 복제돼 있다. 그러다 디자인이 바뀌면 — 버튼 색을 바꾸거나 폰트를 조정하는 작은 변경도 — 모든 화면을 열어서 하나씩 수정해야 한다.

"찾아 바꾸기 하면 되지 않나요?"라는 의문이 생긴다. 값이 완전히 동일할 때는 된다. 하지만 하드코딩이 쌓이면 같은 의도인데 표현이 조금씩 다른 경우가 생긴다. `#3B82F6`이기도 하고 `rgb(59, 130, 246)`이기도 하고 `blue-500`이기도 하다. 전부 같은 파란색이다.

**결국 하드코딩의 문제는 변경 비용이 코드베이스 전체에 분산된다는 것이다.**

---

## 요리로 이해하는 3계층 구조

코드 구조 얘기가 추상적으로 느껴지면 요리로 생각해보자.

매번 요리할 때마다 소금을 얼마나 넣을지 처음부터 고민하고, 간장 브랜드를 그때그때 다르게 쓴다고 생각해보자. 음식이 매번 다른 맛이 날 것이다. 그래서 좋은 요리사는 자기만의 기본 레시피를 가지고 있다. 소스를 미리 만들어두고, 메뉴마다 꺼내서 조합한다.

3계층 구조도 같은 원리다.

> - **1계층 — 디자인 시스템**: 색상, 폰트, 간격, 그림자 등 기본 재료를 표준화한다. "파란색은 항상 이 값"을 한 곳에서 정의한다.
> - **2계층 — 컴포넌트**: 디자인 시스템의 재료를 조합해서 반조리 식품을 만든다. 버튼, 입력창, 카드, 모달 같은 재사용 가능한 단위다.
> - **3계층 — 화면**: 만들어진 컴포넌트를 조합해서 최종 요리를 완성한다. 직접 재료를 손질하지 않는다.

[이미지: 3계층 구조 다이어그램 — 디자인 시스템(재료) → 컴포넌트(반조리) → 화면(완성 요리)]

---

## 1계층: 디자인 시스템 — 재료를 표준화한다

디자인 시스템은 "우리 서비스에서 쓰는 기본값들"을 한 곳에 모은 것이다. 색상 팔레트, 타이포그래피 규칙, 간격 단위, 그림자 강도.

가장 중요한 건 이름을 붙이는 것이다. `#3B82F6`이 아니라 `primary-blue`. `16px`이 아니라 `font-size-body`. 값이 바뀌어도 이름은 유지된다.

```css
/* 전: 여기저기 흩어진 하드코딩 */
color: #3B82F6;
font-size: 16px;
padding: 12px 16px;

/* 후: 디자인 시스템 토큰 */
color: var(--color-primary);
font-size: var(--font-size-body);
padding: var(--spacing-3) var(--spacing-4);
```

**이름을 붙이는 순간 "무엇"이 아니라 "왜"가 코드에 담긴다.** `#3B82F6`은 아무 맥락이 없지만 `--color-primary`는 "이 색이 우리 브랜드 주색이다"는 의미가 있다.

---

## 2계층: 컴포넌트 — 재사용 단위를 만든다

컴포넌트는 디자인 시스템의 토큰을 써서 만든 재사용 가능한 UI 조각이다. 버튼 하나를 만들면 앱 전체에서 쓴다.

중요한 건 컴포넌트가 자기 스타일을 직접 결정하지 않는다는 것이다. 색상은 디자인 시스템에서, 크기는 props로 받는다. 컴포넌트는 조합 규칙만 가진다.

```jsx
// 전: 화면마다 버튼 코드가 복제됨
<button style={{ backgroundColor: '#3B82F6', padding: '12px 16px', borderRadius: '8px' }}>
  확인
</button>

// 후: 컴포넌트 하나를 화면마다 가져다 씀
<Button variant="primary" size="md">확인</Button>
```

버튼 디자인을 바꾸고 싶으면 `Button` 컴포넌트 하나만 수정하면 된다. 37개 화면이 자동으로 바뀐다.

**컴포넌트의 가치는 재사용이 아니라 변경이 한 곳에서 일어난다는 것이다.**

---

## 3계층: 화면 — 조합만 한다

3계층에 도달하면 화면 코드가 드라마틱하게 줄어든다.

기존에 화면 파일이 300줄이었다면, 리팩토링 후에는 50줄 이하가 되는 경우도 많다. 나머지는 전부 컴포넌트 안으로 들어갔다. 화면은 "어떤 컴포넌트를 어떤 순서로 배치할지"만 결정한다.

```jsx
// 전: 화면 파일에 모든 스타일과 로직이 뒤섞임
// 후: 화면은 조합만 한다
function ProfileScreen() {
  return (
    <Screen>
      <Header title="내 프로필" />
      <Avatar src={user.photo} size="lg" />
      <UserInfo name={user.name} email={user.email} />
      <Button variant="outline" onPress={handleEdit}>편집</Button>
    </Screen>
  );
}
```

화면을 읽으면 이 화면이 뭘 하는지 바로 보인다. 스타일 코드가 없어서 구조가 명확하다.

**화면 코드가 짧아진다는 건 각 역할이 제자리를 찾았다는 신호다.**

---

## 리팩토링을 시작하는 순서

이미 하드코딩된 코드를 3계층으로 분리하는 건 한 번에 하려고 하면 무너진다. 작업 순서가 중요하다.

> 1. **디자인 시스템부터**: 색상, 폰트, 간격을 토큰으로 추출한다. 값을 바꾸지 말고 이름만 붙인다.
> 2. **가장 많이 반복되는 것부터 컴포넌트화**: 버튼, 입력창처럼 화면마다 등장하는 것을 먼저 뺀다.
> 3. **화면은 마지막에**: 컴포넌트가 준비된 화면부터 차례로 정리한다.

한 번에 모든 화면을 건드리려 하지 않는다. 새로 만드는 화면부터 3계층 구조로 쓰고, 기존 화면은 건드릴 일이 생길 때 정리하는 식으로 점진적으로 전환하는 게 현실적이다.

---

## 정리

하드코딩은 처음엔 빠르다. 하지만 두 번째로 건드릴 때 그 비용을 치른다.

3계층 구조는 그 비용을 미리 한 곳에 모아두는 작업이다. 처음 만들 때는 조금 더 시간이 걸린다. 하지만 색상 하나를 바꿀 때 파일 37개를 열지 않아도 된다.

**좋은 구조는 변경을 쉽게 만든다.** 레시피를 만들어두는 요리사가 매번 처음부터 고민하는 요리사보다 더 많은 메뉴를 일관된 맛으로 낼 수 있는 것처럼.
