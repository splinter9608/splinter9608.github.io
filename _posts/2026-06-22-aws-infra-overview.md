서버 하나 띄우면 서비스가 되는 거 아닌가. 처음엔 그렇게 생각하기 쉽다.

근데 실제로 서비스를 운영하려면 서버 말고도 꽤 많은 것들이 필요하다. 트래픽을 나눠줄 무언가, 파일을 저장할 곳, 전 세계 사용자에게 빠르게 전달할 방법, 서버가 죽으면 자동으로 살릴 장치.

**AWS Infra & Compute는 그 모든 것을 제공하는 레고 세트다.** 블록 하나하나는 단순하지만, 조합하면 어떤 규모의 서비스도 만들 수 있다.

이번 글에서는 02_Infra & Compute 섹션에서 다룰 서비스들이 전체 그림에서 어떤 역할을 하는지 살펴봅니다.

---

## 도시를 짓는다고 생각해보자

AWS 인프라를 도시에 비유하면 각 서비스의 역할이 선명해진다.

사람들이 살 수 있는 도시를 만들려면 건물만 있어선 안 된다. 도로 표지판, 교통 정리, 창고, 전국 배송망, 필요할 때 임시로 빌리는 공간 — 이 모든 것이 있어야 도시가 돌아간다.

AWS 인프라도 마찬가지다.

---

## 서비스 지도

### EC2 — 건물(서버)

> **EC2(Elastic Compute Cloud)** — AWS에서 빌려 쓰는 가상 서버. CPU, 메모리, OS를 선택해서 내 서버처럼 쓴다.

도시의 건물이다. 사람들이 실제로 머무는 공간. 서비스 로직이 돌아가는 곳이다. 크기도 선택할 수 있다. 작은 단칸방(t2.micro)부터 초대형 빌딩(고사양 인스턴스)까지.

건물이 하나면 사람이 몰릴 때 못 버틴다. 그래서 **ASG(Auto Scaling Group)** 가 등장한다. 사람이 몰리면 건물을 자동으로 더 짓고, 빠지면 철거한다.

### ELB — 교통 정리 안내데스크

> **ELB(Elastic Load Balancer)** — 들어오는 트래픽을 여러 서버에 자동으로 분산시켜주는 서비스.

건물이 여러 채라면 사람들이 어느 건물로 가야 할지 알아야 한다. ELB가 입구에서 "이쪽으로 가세요, 저쪽으로 가세요"를 해준다. HA & Scalability 편에서 직접 만들어봤던 ALB가 여기에 속한다.

### Route 53 — 도로 표지판

> **Route 53** — AWS의 DNS 서비스. 도메인과 IP를 연결하고, 트래픽을 어디로 보낼지 결정한다.

DNS 편에서 다뤘던 그 전화번호부를 AWS에서 관리하는 서비스다. 도메인을 사서 Route 53에 연결하면, 내 서버 주소를 전 세계에 알릴 수 있다.

### CloudFront — 전국 배달망

> **CloudFront** — AWS의 CDN(Content Delivery Network). 전 세계 엣지 서버에 콘텐츠를 캐싱해서 가까운 곳에서 빠르게 제공한다.

서울 서버에서 미국 사용자에게 데이터를 보내면 느리다. CloudFront는 전국(전 세계)에 지점을 두고, 자주 요청되는 콘텐츠를 가장 가까운 지점에 미리 채워둔다. 캐시 편에서 다룬 그 개념이 글로벌 규모로 적용된 것이다.

### EBS — 건물에 붙은 창고

> **EBS(Elastic Block Store)** — EC2 인스턴스에 연결하는 네트워크 드라이브. 인스턴스가 꺼져도 데이터가 남는다.

건물(EC2) 옆에 붙어 있는 창고다. 그 건물 전용이다. 빠르고, 인스턴스가 재시작돼도 데이터가 유지된다. 운영체제, 애플리케이션 데이터를 여기 저장한다.

### EFS — 여러 건물이 함께 쓰는 공유 창고

> **EFS(Elastic File System)** — 여러 EC2 인스턴스가 동시에 접근할 수 있는 공유 파일 시스템.

EBS가 건물 전용 창고라면, EFS는 여러 건물이 공동으로 쓰는 창고다. EC2 인스턴스 여러 대가 같은 파일을 동시에 읽고 쓸 수 있다.

### S3 — 대형 물류 창고

> **S3(Simple Storage Service)** — 거의 무한대 용량의 객체 스토리지. 파일, 이미지, 백업, 로그 등을 저장한다.

EBS/EFS가 서버에 붙어 있는 창고라면, S3는 도시 외곽의 거대한 물류 창고다. 사실상 용량 제한이 없다. 이미지, 동영상, 백업 파일, 정적 웹사이트 파일 등을 저장하는 데 쓴다.

### Containers — 조립식 건물

> **컨테이너(ECS/EKS/Fargate)** — 애플리케이션과 실행 환경을 묶어서 어디서든 동일하게 실행할 수 있는 패키지. 자세한 내용은 나중에 다룹니다.

EC2가 건물이라면 컨테이너는 조립식 건물이다. 어느 땅(서버)에서든 똑같이 조립해서 쓸 수 있다. Docker를 써봤다면 그게 컨테이너다.

### Serverless — 공유 오피스

> **서버리스(Lambda 등)** — 서버를 직접 관리하지 않고, 코드 실행 시간만큼만 비용을 내는 방식. 자세한 내용은 나중에 다룹니다.

건물을 소유하지 않고, 필요할 때만 공유 오피스 자리를 빌리는 것과 같다. 서버를 24시간 켜두지 않아도 된다. 요청이 올 때만 코드가 실행되고, 끝나면 사라진다.

## 전체 그림

[이미지: AWS 인프라 서비스 관계도 — 사용자 → Route53 → CloudFront → ELB → EC2(ASG) → EBS/EFS/S3]

사용자가 내 서비스에 접속하는 흐름을 따라가면:

> 1. 사용자가 도메인 입력 → **Route 53**이 IP로 변환
> 2. 요청이 **CloudFront** 엣지에 도달 → 캐시에 있으면 바로 응답
> 3. 없으면 **ELB**로 전달 → 여러 EC2 중 하나로 라우팅
> 4. **EC2**가 요청 처리 → **EBS/EFS/S3**에서 필요한 데이터 꺼냄
> 5. 응답 반환

이 흐름을 이해하면, 앞으로 각 서비스를 배울 때 "이게 전체 그림에서 어디에 있는 건지"가 보인다.

## 그 밖의 스토리지 도구들

위의 서비스들이 대부분의 케이스를 커버한다. 그런데 AWS에는 좀 더 특수한 상황을 위한 스토리지 관련 도구들도 있다. 자세히 다루지는 않고, 이런 것들이 있다는 정도만 알아두자.

- **Snow Family (Snowcone/Snowball)** — 인터넷으로 옮기기엔 너무 큰 데이터를 물리 장비에 담아 택배로 AWS에 보내는 서비스. 페타바이트 단위 마이그레이션에 쓴다.
- **FSx** — Windows 서버용(FSx for Windows), 고성능 컴퓨팅용(FSx for Lustre) 등 특수 목적 파일 시스템. EFS가 일반 Linux 환경이라면, FSx는 특정 환경에 최적화된 버전이다.
- **Storage Gateway** — 온프레미스 서버와 AWS 스토리지를 연결하는 브리지. 클라우드로 완전히 이전하지 않고 일부만 AWS를 쓰는 하이브리드 환경에서 쓴다.
- **Transfer Family** — FTP/SFTP 프로토콜로 S3나 EFS에 파일을 전송하는 서비스. 레거시 FTP 기반 워크플로를 AWS로 마이그레이션할 때 쓴다.
- **DataSync** — 온프레미스에서 AWS로, 또는 AWS 서비스 간 대량 데이터를 자동으로 동기화하는 서비스.

SAA 시험에 개념으로 나오지만, 일반적인 웹 서비스 구축에서는 접할 일이 드물다. 필요할 때 찾아보는 것으로 충분하다.
