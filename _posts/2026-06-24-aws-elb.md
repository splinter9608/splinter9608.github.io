서버가 여러 대 있으면 트래픽이 분산된다. 맞는 말이다.

틀린 건 "알아서"라는 부분이다. 서버 10대를 띄워놓는다고 트래픽이 알아서 나눠지지 않는다. 누군가 입구에 서서 "이쪽으로 가세요, 저쪽으로 가세요"를 해줘야 한다. 그리고 그게 단순한 역할처럼 보여도, 실제로는 꽤 정교한 판단을 한다.

**ELB(Elastic Load Balancer)는 그 판단을 하는 안내데스크다.** 어느 서버가 살아있는지 확인하고, 들어오는 요청의 성격에 따라 적절한 서버로 보낸다.

이번 글에서는 1) ELB가 실제로 하는 일, 2) ALB와 NLB의 차이와 선택 기준, 3) 라우팅 규칙과 타겟 그룹 개념을 살펴봅니다.

---

## 안내데스크가 하는 두 가지 일

ELB는 두 가지 일을 동시에 한다.

첫 번째는 **분산**이다. 들어오는 요청을 여러 서버에 나눠서 보낸다. 한 서버에 몰리지 않게.

두 번째가 더 중요하다. **헬스 체크**다. ELB는 주기적으로 각 서버에 "잘 있어요?" 신호를 보낸다. 응답이 없거나 오류가 오면 그 서버를 안내 목록에서 뺀다. 죽은 서버로 요청을 보내지 않는다.

**헬스 체크가 없는 로드밸런서는 절반짜리다.** 서버가 죽었는데도 계속 거기로 요청을 보내는 안내데스크는 없느니만 못하다.

[이미지: ELB 헬스 체크 흐름 — 정상 서버(초록), 비정상 서버(빨강) 분리]

## 안내데스크도 종류가 있다 — ALB vs NLB

ELB는 단일 제품이 아니다. AWS에는 세 종류의 로드밸런서가 있고, 각각 쓰임새가 다르다.

> - **ALB** (Application Load Balancer) — HTTP/HTTPS를 이해하는 안내데스크
> - **NLB** (Network Load Balancer) — 내용은 안 보고 무조건 빠르게 통과시키는 VIP 라인
> - **GWLB** (Gateway Load Balancer) — 방화벽, 패킷 검사 장비를 거치게 하는 특수 목적용. 자세한 내용은 나중에 다룹니다.

대부분의 웹 서비스에서 쓰는 건 ALB다. NLB는 성능이 극단적으로 중요하거나, HTTP가 아닌 프로토콜을 써야 할 때 꺼낸다.

### ALB — 내용을 보고 판단하는 안내데스크

ALB는 HTTP 요청의 내용을 읽는다. URL 경로가 `/api/`면 API 서버로, `/images/`면 이미지 서버로, 나머지는 메인 서버로 — 이런 판단이 가능하다. 같은 도메인이라도 요청의 성격에 따라 다른 서버로 보낼 수 있다는 뜻이다.

> ALB가 볼 수 있는 것들:
> - **경로 기반**: `/api/*` → API 서버, `/static/*` → 정적 파일 서버
> - **호스트 기반**: `api.example.com` → API 서버, `www.example.com` → 웹 서버
> - **헤더/쿼리**: 특정 헤더 값이나 쿼리 파라미터에 따라 분기

**ALB 하나로 여러 서비스를 도메인이나 경로로 나눠서 운영할 수 있다.** 서버를 도메인마다 따로 노출시킬 필요가 없다.

[이미지: ALB 경로 기반 라우팅 — /api/* → API 서버군, /app/* → 앱 서버군 분기]

### NLB — 내용은 안 보고 그냥 빠르게

NLB는 TCP/UDP 레벨에서 동작한다. HTTP가 뭔지 모른다. URL 경로도, 헤더도 보지 않는다. 그냥 들어온 패킷을 서버로 던진다.

대신 빠르다. ALB보다 지연시간이 낮고, 초당 수백만 건의 요청도 처리한다. 고정 IP를 제공해서 방화벽 설정이 필요한 환경(특정 IP만 허용하는 기업 네트워크 등)에서도 쓸 수 있다.

**HTTP가 아닌 프로토콜을 써야 하거나, 지연시간이 극도로 낮아야 할 때 NLB를 선택한다.** 일반 웹 서비스라면 ALB가 맞다.

## 어디로 보낼지 목록을 관리한다 — 타겟 그룹

ALB가 "이 경로는 API 서버로"라고 결정했다면, 다음 질문은 "API 서버가 어떤 서버들이야?"다. 이 목록을 **타겟 그룹(Target Group)**이 관리한다.

> **타겟 그룹** — ELB가 트래픽을 보낼 목적지들의 묶음. EC2 인스턴스, IP 주소, 또는 Lambda 함수가 타겟이 될 수 있다.

구조로 보면:

> ELB → 라우팅 규칙 → 타겟 그룹 → 개별 서버들

라우팅 규칙이 "어디 그룹으로 보낼지"를 정하고, 타겟 그룹 안에서 실제 분산이 일어난다. 헬스 체크도 타겟 그룹 단위로 설정한다.

타겟 그룹을 별도로 두는 이유는 유연성 때문이다. 서버를 교체하거나 늘릴 때 라우팅 규칙은 건드리지 않고 타겟 그룹 안의 목록만 바꾸면 된다.

## 서버가 자동으로 늘어도 안내는 끊기지 않는다 — ASG 연동

[앞선 EC2 글]({% post_url 2026-06-23-aws-ec2 %})에서 ASG(Auto Scaling Group)를 다뤘다. ASG가 트래픽에 따라 서버를 자동으로 늘리고 줄인다는 것.

근데 새 서버가 생겼을 때 ELB가 그걸 어떻게 알까?

ASG와 ELB의 타겟 그룹을 연결해두면 자동으로 해결된다. ASG가 새 인스턴스를 만들면 자동으로 타겟 그룹에 등록된다. 헬스 체크를 통과하면 바로 트래픽을 받는다. 서버가 줄어들 때는 진행 중인 요청을 다 처리한 뒤에 타겟 그룹에서 빠진다(드레이닝).

**ELB + ASG 조합이 완성되면, 트래픽 변화에 사람이 개입할 일이 없다.** 서버는 자동으로 늘고, 안내는 자동으로 갱신된다.

---

## 정리

ELB는 단순한 트래픽 분산기가 아니다. 어떤 서버가 살아있는지 판단하고, 요청의 성격에 따라 적절한 곳으로 보내고, 서버 목록이 바뀌어도 자동으로 따라간다.

**건물(EC2)을 아무리 많이 지어도, 안내데스크(ELB) 없이는 사람들이 어디로 가야 할지 모른다.** 그리고 좋은 안내데스크는 손님이 도착하기 전에 이미 어느 층이 붐비는지, 어느 층이 문을 닫았는지 알고 있다.

안내데스크까지 세웠으면, 이제 전 세계 손님이 빠르게 들어올 수 있도록 도로망을 깔 차례다.
