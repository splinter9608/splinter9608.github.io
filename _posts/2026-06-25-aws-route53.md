도메인을 샀으면 서버 IP 연결하면 끝이다. 맞는 말이다.

근데 그게 전부라면 Route 53이 AWS의 핵심 서비스 중 하나일 이유가 없다. 단순히 도메인과 IP를 연결하는 건 무료 DNS 서비스로도 된다.

**Route 53이 하는 일은 "어디로 연결할지"가 아니라 "누구에게, 언제, 어떻게 연결할지"를 결정하는 것이다.** 서울 사용자는 서울 서버로, 뉴욕 사용자는 버지니아 서버로, 메인 서버가 죽으면 백업 서버로 — 이런 판단이 DNS 레벨에서 일어난다.

이번 글에서는 1) Route 53의 레코드 타입, 2) AWS 리소스 연결에만 쓰는 Alias 레코드, 3) 상황별 라우팅 정책을 살펴봅니다. DNS 기본 개념은 [앞선 글]({% post_url 2026-06-20-aws-dns-network %})에서 다뤘으니 여기선 넘어갑니다.

## 레코드 타입: 표지판의 종류

도로 표지판도 종류가 있다. "직진", "우회전", "저 건물 이름은 XX빌딩". Route 53의 레코드 타입도 마찬가지다. 무엇을 가리키느냐에 따라 타입이 다르다.

> - **A 레코드** — 도메인 → IPv4 주소. 가장 기본. `example.com → 1.2.3.4`
> - **AAAA 레코드** — 도메인 → IPv6 주소
> - **CNAME 레코드** — 도메인 → 다른 도메인. `www.example.com → example.com`
> - **MX 레코드** — 이메일 서버 지정. `@example.com` 으로 오는 메일을 어디로 보낼지
> - **NS 레코드** — 이 도메인의 DNS를 관리하는 서버가 어디인지
> - **TXT 레코드** — 도메인 소유자 인증, SPF 설정 등 텍스트 정보

일상에서 가장 많이 만지는 건 A와 CNAME이다. 그리고 AWS에서만 쓸 수 있는 타입이 하나 더 있다.

## A 레코드로 안 되는 상황 — Alias 레코드

"ALB에 도메인을 연결하고 싶어요."

이때 A 레코드를 쓰려고 하면 막힌다. ALB는 고정 IP가 없다. DNS 이름(`web-alb-xxx.ap-northeast-2.elb.amazonaws.com`)만 있다. IP가 없으니 A 레코드를 쓸 수 없다.

CNAME을 쓰면 되지 않냐고 생각할 수 있다. 기술적으로 가능하지만 두 가지 문제가 있다. 첫째, CNAME은 루트 도메인(`example.com`)에 쓸 수 없다. `www.example.com`엔 되지만 `example.com` 자체엔 안 된다. 둘째, DNS 조회가 한 단계 더 일어나서 느리다.

**Alias 레코드는 이 두 문제를 동시에 해결한다.** Route 53에서만 쓸 수 있는 타입으로, AWS 리소스를 직접 가리킨다. 루트 도메인에도 쓸 수 있고, IP 조회 없이 바로 연결된다. 요금도 없다.

> Alias 레코드로 연결할 수 있는 AWS 리소스:
> - ALB / NLB / CLB
> - CloudFront 배포
> - S3 정적 웹사이트
> - API Gateway
> - 같은 호스팅 영역의 다른 Route 53 레코드

**AWS 리소스에 도메인을 연결할 때는 항상 Alias를 쓴다.** CNAME보다 빠르고, 루트 도메인에도 쓸 수 있고, 추가 비용도 없다.

## 표지판에 조건을 붙인다 — 라우팅 정책

단순히 "이쪽으로 가세요"가 아니라 조건에 따라 다른 방향을 가리키는 표지판. 이게 Route 53 라우팅 정책이다.

### 단순 라우팅 (Simple)

조건 없다. 도메인 → 하나의 주소. 가장 기본이고, 헬스 체크와 연동되지 않는다.

서버가 하나일 때 쓴다. 죽어도 Route 53은 모른다.

### 가중치 라우팅 (Weighted)

같은 도메인에 여러 레코드를 만들고, 각각 가중치를 준다.

> `example.com` → 서버 A (가중치 70) / 서버 B (가중치 30)

100명이 접속하면 70명은 A로, 30명은 B로 간다. **블루/그린 배포나 카나리 테스트에 쓴다.** 새 버전 서버를 10% 가중치로 올려두고, 문제 없으면 점진적으로 늘린다.

### 지연시간 라우팅 (Latency)

사용자 위치를 보지 않는다. 어떤 리전이 더 빠른지를 본다. AWS가 전 세계 리전 간 지연시간 데이터를 가지고 있어, 요청자에게 가장 빠른 리전으로 보내준다.

**글로벌 서비스에서 가장 먼저 고려하는 정책이다.** 서울 사용자는 ap-northeast-2로, 뉴욕 사용자는 us-east-1로 자동으로 간다.

### 장애조치 라우팅 (Failover)

메인 서버와 백업 서버를 지정한다. 평소엔 메인으로, 헬스 체크가 실패하면 자동으로 백업으로 전환된다.

> Primary: 메인 서버 (헬스 체크 연동)
> Secondary: 백업 서버 (또는 S3 정적 오류 페이지)

**DNS 레벨 장애조치라 빠르다.** 서버가 죽으면 TTL이 지난 후 모든 새 요청이 백업으로 간다. 애플리케이션 코드를 건드리지 않아도 된다.

### 지리적 라우팅 (Geolocation)

지연시간이 아니라 사용자의 실제 위치를 본다. 한국에서 접속하면 한국 서버로, 유럽에서 접속하면 유럽 서버로.

지연시간 라우팅과 헷갈리기 쉽다. 차이는 이렇다:

> - **지연시간 라우팅**: 어디서 접속하든 "가장 빠른 서버"로
> - **지리적 라우팅**: 한국 사용자면 "한국 서버"로 (빠름과 무관하게)

지역별로 다른 콘텐츠를 보여줘야 할 때 (언어, 법규, 가격 등) 지리적 라우팅을 쓴다.

## 표지판이 살아있는지 확인한다 — 헬스 체크

장애조치 라우팅이 제대로 작동하려면 Route 53이 서버 상태를 알아야 한다. 이게 **헬스 체크**다.

Route 53은 주기적으로 지정한 엔드포인트에 HTTP 요청을 보낸다. 응답이 오면 정상, 안 오거나 오류 코드면 비정상으로 판단한다.

> 헬스 체크 설정 예시:
> - 프로토콜: HTTP
> - 경로: `/health`
> - 인터벌: 30초
> - 실패 임계값: 3회 (3번 연속 실패 시 비정상 처리)

비정상으로 판단되면 해당 레코드로 트래픽을 보내지 않는다. 장애조치 라우팅과 결합하면 서버 장애를 사람이 개입하지 않아도 자동으로 우회할 수 있다.

**헬스 체크 없는 장애조치는 작동하지 않는다.** Route 53이 서버 상태를 모르면 판단 기준이 없다.

## 정리

도메인 연결은 IP를 붙이는 것보다 훨씬 정교한 작업이다. 누가 접속하느냐, 어떤 서버가 살아있느냐, 어떤 비율로 트래픽을 나눌 것이냐 — 이 모든 결정이 Route 53에서 일어난다.

도로 표지판은 원래 단순하다. 거리와 방향만 알려준다. 그런데 Route 53의 표지판은 날씨를 보고, 도로 상태를 보고, 운전자의 출발지를 보고 방향을 바꾼다. 도메인 하나에 이렇게 많은 판단이 담길 수 있다.

도로망까지 완성됐으면, 이제 전국 어디서나 빠르게 배달될 수 있도록 물류망을 깔 차례다.
