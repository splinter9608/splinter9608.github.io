## 여러 구름이 서로 받쳐주는 구조, 직접 만들어봅시다

[앞선 글]({% post_url 2026-06-18-aws-ha-scalability %})에서 얘기했던 것처럼, 구름 하나가 버텨주는 구조로는 언젠가 한계가 온다. 이번엔 냄비 두 개를 다른 선반에 올려두는 것처럼 — EC2 두 대를 서로 다른 AZ에 띄우고, ELB로 묶어봅시다.

---

## 이 페이지에서 할 것

EC2(AWS의 가상 서버) 두 대를 서로 다른 가용 영역(AZ)에 배치하고, ELB(트래픽을 나눠주는 창구)로 묶어봅니다. 하나가 죽어도 나머지가 트래픽을 받는 구조를 직접 구성해봅니다.

> EC2, ELB는 나중에 자세히 다룰 예정입니다. 여기서는 "서버"와 "트래픽 분배기" 정도로 이해하고 따라오시면 됩니다.

---

## 준비물

- AWS 계정 (프리 티어 가능)
- 같은 리전 안에 VPC와 서브넷 2개 이상 (기본 VPC 그대로 써도 됩니다)
  - **VPC** — AWS 안의 내 전용 네트워크 공간입니다. **서브넷**은 그 안을 AZ별로 나눈 구역이에요. 둘 다 나중에 자세히 다룹니다.

---

## Step 1 — EC2 인스턴스 2대를 서로 다른 AZ에 띄우기

EC2는 AWS에서 빌려 쓰는 가상 서버입니다. 자세한 내용은 나중에 다루고, 지금은 "내 서버 하나"라고 생각하면 됩니다.

Multi-AZ의 핵심은 물리적으로 분리된 건물에 두는 것입니다. 같은 AZ에 두면 의미가 없습니다.

**첫 번째 인스턴스 (AZ-a)**

콘솔 경로: `서비스 > EC2 > 인스턴스 > 인스턴스 시작`

설정값:
```
이름: web-server-az-a
AMI: Amazon Linux 2023 (프리 티어 사용 가능)
인스턴스 유형: t2.micro
가용 영역: ap-northeast-2a (서울 리전 기준)
보안 그룹: HTTP(80), SSH(22) 인바운드 허용
```

- **AMI(Amazon Machine Image)** — 서버를 시작할 때 쓰는 OS + 기본 소프트웨어가 담긴 이미지입니다. 스마트폰 공장 초기화 이미지와 비슷합니다.
- **인스턴스 유형** — 서버 사양입니다. t2.micro는 CPU 1개, 메모리 1GB로 가장 작은 사양이지만 프리 티어에서 무료로 쓸 수 있습니다.
- **보안 그룹** — AWS의 방화벽입니다. 어떤 트래픽을 허용할지 규칙을 정합니다. **인바운드**는 외부에서 서버로 들어오는 트래픽을 말합니다.

고급 세부 정보 > 사용자 데이터에 아래 붙여넣기:
```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>AZ-A 서버입니다</h1>" > /var/www/html/index.html
```

- **사용자 데이터(User Data)** — 인스턴스가 처음 시작될 때 자동으로 실행되는 스크립트입니다. 매번 직접 설정하는 대신 이걸로 자동화합니다.
- **httpd** — Apache 웹 서버입니다. 브라우저 요청을 받아 HTML 파일을 돌려주는 소프트웨어로, 나중에 자세히 다룹니다.

**두 번째 인스턴스 (AZ-b)**

같은 방법으로 하나 더 만들되, 가용 영역만 다르게:
```
이름: web-server-az-b
가용 영역: ap-northeast-2b
사용자 데이터: "<h1>AZ-B 서버입니다</h1>" 로 변경
```

2분 정도 기다리면 두 인스턴스 모두 `실행 중` 상태가 됩니다.

```
예상 결과:
web-server-az-a | 실행 중 | ap-northeast-2a | 퍼블릭 IP: xxx.xxx.xxx.xxx
web-server-az-b | 실행 중 | ap-northeast-2b | 퍼블릭 IP: yyy.yyy.yyy.yyy
```

각 퍼블릭 IP로 브라우저에서 접속해보면 "AZ-A 서버입니다" / "AZ-B 서버입니다"가 각각 뜨면 됩니다.

---

## Step 2 — 대상 그룹(Target Group) 만들기

ALB(Application Load Balancer)는 들어오는 요청을 여러 서버에 나눠주는 역할을 합니다. 자세한 내용은 ELB 편에서 다룰 예정이고, 지금은 "트래픽 분배기"라고 이해하면 됩니다.

대상 그룹은 그 분배기가 어디로 보낼지 알 수 있게 서버 목록을 등록해두는 곳입니다.

콘솔 경로: `서비스 > EC2 > 로드 밸런싱 > 대상 그룹 > 대상 그룹 생성`

```
대상 유형: 인스턴스
대상 그룹 이름: web-tg
프로토콜: HTTP / 포트: 80
VPC: 기본 VPC (인스턴스와 같은 VPC)
상태 확인 경로: /
```

다음 단계에서 `web-server-az-a`, `web-server-az-b` 두 인스턴스를 모두 선택해서 추가합니다.

```
대상 등록 확인:
web-server-az-a | 80 | ap-northeast-2a
web-server-az-b | 80 | ap-northeast-2b
```

---

## Step 3 — Application Load Balancer(ALB) 만들기

이제 두 서버 앞에 트래픽을 나눠주는 창구를 만듭니다. ALB가 여기서 하는 역할은 단순합니다 — 요청이 들어오면 AZ-a 서버와 AZ-b 서버에 번갈아 넘겨주는 것입니다.

콘솔 경로: `서비스 > EC2 > 로드 밸런싱 > 로드 밸런서 > 로드 밸런서 생성`

```
유형: Application Load Balancer
이름: web-alb
체계: 인터넷 경계
IP 주소 유형: IPv4

네트워크 매핑:
  - ap-northeast-2a 서브넷 선택
  - ap-northeast-2b 서브넷 선택  ← 두 AZ 모두 선택해야 합니다

보안 그룹: HTTP(80) 인바운드 허용하는 그룹

리스너:
  프로토콜: HTTP / 포트: 80
  기본 작업: web-tg 로 전달
```

- **리스너** — ALB가 어떤 포트/프로토콜로 들어오는 요청을 받을지 정의하는 규칙입니다. 여기선 80번 포트(HTTP)로 들어오는 요청을 대상 그룹으로 넘기도록 설정합니다.

생성 후 `활성` 상태가 될 때까지 2~3분 대기합니다.

---

## Step 4 — 연결 확인

ALB가 활성화되면 DNS 이름이 발급됩니다.

- **DNS 이름** — 숫자 IP 대신 문자로 된 주소입니다. DNS가 어떻게 작동하는지는 나중에 따로 다룹니다.

콘솔에서 확인:
```
로드 밸런서 > web-alb > DNS 이름
예: web-alb-1234567890.ap-northeast-2.elb.amazonaws.com
```

브라우저에서 이 DNS 주소로 접속하면 새로고침할 때마다 AZ-A / AZ-B 서버가 번갈아 응답하는 것을 볼 수 있습니다.

```
첫 번째 접속: AZ-A 서버입니다
새로고침: AZ-B 서버입니다
새로고침: AZ-A 서버입니다
```

**P.S.** 번갈아 나오지 않더라도 괜찮습니다. ALB는 라운드 로빈(요청을 순서대로 번갈아 분배하는 방식)이 기본이지만 세션 캐시 때문에 같은 서버로 몇 번 붙는 경우도 있습니다.

---

## 이렇게 됐으면 됩니다

아래 세 가지가 모두 확인되면 구성 완료입니다.

- [ ] EC2 인스턴스 2대가 서로 다른 AZ(2a, 2b)에서 `실행 중` 상태
- [ ] 대상 그룹에서 두 인스턴스 모두 `정상(Healthy)` 표시
- [ ] ALB DNS로 접속 시 페이지가 뜨고, AZ-A / AZ-B 응답이 확인됨

[이미지: EC2 콘솔 - 두 인스턴스가 각각 2a, 2b에서 실행 중인 화면]

[이미지: 대상 그룹 - 두 타겟 모두 Healthy 상태]

---

## HA 확인 (선택)

하나가 죽어도 되는지 실제로 테스트해볼 수 있습니다.

EC2 콘솔에서 `web-server-az-a`를 선택 → `인스턴스 상태 > 중지`

30초 후 ALB DNS로 다시 접속하면 `AZ-B 서버입니다`만 응답합니다. AZ-A가 죽었지만 서비스는 살아있는 상태입니다. 이게 HA입니다.

테스트 후엔 az-a 인스턴스를 다시 시작하면 됩니다.
